  * **Branch 2: 웹 서버 엔진 구동 및 다중 사용자 보안 권한 통제**

    * 파일 소유권 등기 이전 명의 변경 (`chown apache:apache`)의 당위성
  * **Branch 3: 커맨드 라인 인터페이스(CLI) 번역기 시스템 이해**
    * Windows 최신 자동화 엔진: **파워쉘 (PowerShell)**의 환경 변수 문법
    * 전 세계 클라우드 표준 인프라: **명령 프롬프트(CMD) 대비 리눅스 터미널의 우위성**
    * 데비안 가문의 절대강자: **우분투(Ubuntu)** 운영체제의 특징 및 아마존 리눅스와의 문법 비교

## 📑 Core Essence: 개념별 매커니즘 심층 분석

### 1. 파일 이사와 리눅스 경로 시스템의 비밀

#### 🛸 파일 수송선 (`scp` - Secure Copy Protocol)
내 로컬 노트북(윈도우)에 저장된 웹사이트 구성 파일 3대장(`index.html`, `style.css`, `app.js`)을, 인터넷 너머 멀리 떨어진 AWS 가상 컴퓨터(EC2 리눅스 소스 서버)로 안전하게 복사하여 전송하는 암호화 수송선 명령어입니다.
* **작동 원리**: 리눅스는 기본적으로 보안을 위해 외부인의 접근을 철저히 통제합니다. 따라서 AWS를 생성할 때 발급받은 마스터키 원본 파일(`-i` 옵션 뒤의 `*.pem` 파일)을 인증서로 제시해야만 수송선이 가상 컴퓨터 내부로 진입할 수 있습니다.
* **최종 수신지**: 기본적으로 외부에서 접속할 때는 보안상 안전한 유저의 개인 방(`ec2-user` 계정의 홈 디렉토리)을 수신 주소로 지정(`~ /`)하여 파일을 안전하게 이사시킵니다.

#### 🏠 홈 디렉토리 (`~`)의 본질
리눅스라는 거대한 공동 아파트 시스템 안에서, 현재 로그인한 사용자 계정(`ec2-user`)에게만 독점적으로 부여된 **'나만의 완벽한 개인 독방 공간'**입니다.
* **격리성과 안정성**: 리눅스는 다중 사용자(Multi-User) 시스템이기 때문에, 한 사람이 시스템 전체를 망가뜨리지 못하도록 철저히 방을 나눕니다. 이 홈 디렉토리 안에서는 최고 관리자 승인 도장(`sudo`)을 받지 않고도 파일을 만들고, 수정하고, 지우는 모든 행위가 100% 자유롭습니다.
* **절대경로 주소**: 시스템 본부 기준 주소는 `/home/ec2-user` 이며, 이를 매번 치기 귀찮기 때문에 리눅스에서는 물결 기호(`~`)라는 단축 심볼을 제공합니다.

#### 🗺️ 절대경로 vs 상대경로의 명확한 차이
리눅스 파일 시스템에서 특정 방(디렉토리)이나 파일의 위치를 알려주는 내비게이션 주소 지정 방식은 두 가지로 나뉩니다.
1. **절대경로**: 내 현재 위치가 서울이든 부산이든 상관없이, 언제나 시스템의 기점인 최상위 총본부(어미 슬래시 `/`)에서부터 모든 방 이름을 순서대로 꼼꼼하게 다 적어주는 '불변의 우편번호 주소' 방식입니다. (예: `/var/www/html/index.html`)
2. **상대경로**: "지금 내가 서 있는 방"을 기준으로 길을 안내하는 방식입니다. 현재 방은 점 하나(`.`), 내 바로 상위 방(부모 방)은 점 두 개(`..`)로 표시합니다. 내가 지금 `/var/www/` 방에 서 있다면, `html/index.html`이라고만 적어도 알아듣는 유연한 방식입니다.

#### 🌍 세계 공통 웹 진열대 (`/var/www/html/`)의 단어별 구조 분석
아파치 웹 서버가 외부 손님에게 웹페이지를 서빙하기 위해 무조건 바라보도록 뇌에 고정되어 있는 주소입니다. 이 긴 이름은 무작위로 지어진 것이 아니라 리눅스 표준 규격에 따른 명확한 유래가 있습니다.
* **`/` (루트)**: 리눅스 컴퓨터의 세포가 시작되는 가장 첫 번째 최상위 본부 공간입니다.
* **`var` (Variable)**: 시스템 운영 중 로그 기록이나 웹 가동 파일처럼, 파일의 내용이나 용량이 자꾸자꾸 추가되고 '변할 수 있는(Variable) 데이터'들을 따로 격리해 놓은 전용 구역입니다.
* **`www` (World Wide Web)**: `var` 구역 안에서 특별히 '인터넷 웹 서비스 관련 프로그램'들만 모여서 활동하도록 지정된 방입니다.
* **`html` (HyperText Markup Language)**: 진짜 웹사이트의 소스 코드 원본 파일(`index.html` 등)들이 최종적으로 안착하여 인터넷 손님들을 맞이하는 '진짜 진열대'입니다.

### 2. 권한(Permission)과 소유권(Ownership) 통제 매커니즘

#### 🔐 보안 자물쇠 비밀번호 (`chmod 644`)의 수학적 비밀
인터넷 손님이 웹브라우저 주소창에 내 IP를 치고 들어왔을 때, 컴퓨터가 "이 파일은 외부인이 볼 수 없는 보안 파일입니다"라며 `403 Forbidden` 에러를 뿜지 않게 하려면 파일 자물쇠의 설정을 변경해야 합니다. 리눅스는 권한을 **[읽기=4], [쓰기=2], [실행=1]**이라는 숫자의 합으로 계산합니다.

`chmod 644`라는 세 자릿수 숫자는 완벽하게 고정된 세 개의 계급 서열을 통제합니다.



1. **첫 번째 자리 [6] - 소유자(Owner) 권한**: 파일을 만든 주인(나)의 권한입니다. `4(읽기) + 2(쓰기) = 6`이므로, 나는 이 웹 소스코드를 언제든 열어보고 코드를 수정하여 저장할 수 있습니다.
2. **두 번째 자리 [4] - 그룹(Group) 권한**: 나와 같은 연구실이나 팀에 속한 팀원들의 권한입니다. `4(읽기)`만 부여되었으므로 팀원들은 코드를 열어볼 수는 있지만 마음대로 수정할 수는 없습니다.
3. **세 번째 자리 [4] - 기타 사용자(Others) 권한**: **여기가 가장 중요합니다.** 전 세계 인터넷을 타고 들어오는 손님들이 바로 이 계급에 속합니다. 이 자리에 최소 `4(읽기)` 권한이 있어야만 외부 웹브라우저가 이 파일을 읽어서 화면에 띄워줄 수 있습니다. 만약 이 숫자가 0이나 2라면 외부인은 절대 내 홈페이지를 볼 수 없습니다.

* **💡 권한 응용 오답 노트**:
  * **`664`**: 만약 협업하는 팀원들에게도 나와 동등하게 코드를 수정할 권리를 주고 싶다면, 두 번째 자리에 쓰기(`2`)를 더해 `664`로 세팅하는 것이 안전합니다.
  * **`614`**: 두 번째 자리에 실행(`1`) 권한만 덜렁 주는 설정입니다. 웹 페이지 파일(`html`)은 프로그램처럼 실행하는 파일이 아닌 텍스트 파일이므로, 읽지도 쓰지도 못하게 막고 실행만 하라는 설정은 리눅스 보안 관점에서 매우 모순적이고 기괴한 오답 세팅이 됩니다.

#### 🪪 명의 등기 이전 (`chown apache:apache`)의 당위성
수송선(`scp`)을 타고 처음 들어온 파일의 주인은 전송자인 `ec2-user` 계정으로 등록되어 있습니다. 하지만 실제 1층 로비에서 서빙을 담당하며 외부 손님에게 웹페이지를 실시간으로 전달하는 주체는 아파치 웹 서버 프로그램, 즉 `apache`라는 가상의 요원 계정입니다.
* **왜 명의를 바꿔야 하나요?**: 주인이 `ec2-user`로 되어 있으면, `apache` 요원이 손님이 올 때마다 남의 방에 있는 파일을 눈치 보며 꺼내와야 하므로 보안 제약이 걸리거나 속도가 지연될 수 있습니다. 
* **구조적 해결**: `chown apache:apache` 명령을 통해 파일의 앞 주인(User)과 뒤 그룹(Group)의 명의를 전부 아파치 요원 앞으로 완전히 등기 이전해 줍니다. 이로써 아파치 요원은 완벽한 정당성과 소유권을 가지고 외부 손님에게 가장 빠르고 안전하게 웹 사이트 서비스를 제공할 수 있게 됩니다.

### 3. 인터페이스 시스템 (PowerShell, Terminal, Ubuntu)의 본질

#### ⚡ 윈도우의 끝판왕 마법 지팡이, 파워쉘(PowerShell)
마이크로소프트가 과거의 유산인 낡은 CMD를 완전히 대체하기 위해 만든, 윈도우 운영체제 통제용 최신 객체지향 쉘 환경입니다.
* **디자인 및 심볼**: 아이콘 배경이 진한 파란색이며 명령어 창임을 뜻하는 `>_` 기호가 새겨져 있습니다. 프로그램을 가동하면 맨 왼쪽에 **`PS`**라는 프롬프트 심볼이 붙어, 구형 CMD 창이 아닌 초강력 파워쉘 엔진이 대기 중임을 작업자에게 증명합니다.
* **환경 변수 문법의 정석**: 내 노트북 안에서 사용자 계정의 핵심 루트 폴더 주소를 불러올 때, 구형 CMD는 `%USERPROFILE%`을 썼지만 파워쉘 전용 정석 문법은 환경 주머니에서 변수를 호출하는 **`$env:USERPROFILE`** (중간에 콜론 `:` 사용 필수)입니다.

#### 🏛️ 현대 인프라 실무 명령어 창의 냉정한 서열
1. **리눅스 터미널 (Bash) [🥇 압도적 1위]**: 구글, 네이버, AWS 등 전 세계 모든 클라우드 서버 인프라의 90% 이상은 오직 글자만 돌아가는 가볍고 안정적인 리눅스 OS를 사용합니다. 따라서 엔지니어들에게 리눅스 터미널 명령어는 선택이 아닌 필수 언어이자 절대 표준입니다.
2. **윈도우 파워쉘 (PowerShell) [🥈 2위]**: 내 개인 개발용 노트북이 윈도우 환경일 때, 리눅스 원격 서버로 파일을 쏘아 올리는 수송선(`scp`)을 기동하거나 내 로컬 환경을 세팅할 때 보조 도구로 강력하게 사용됩니다.
3. **윈도우 명령 프롬프트 (CMD) [🥉 3위]**: 1980년대 MS-DOS 시절 구조를 그대로 안고 있어 현대의 복잡한 클라우드 명령어들을 전혀 알아듣지 못합니다. 간단한 내 컴퓨터 IP 확인(`ipconfig`) 목적 외에는 실무에서 거의 은퇴 수순을 밟은 낡은 도구입니다.

#### 🐧 리눅스 가문의 친절한 대장, 우분투(Ubuntu)
리눅스 커널(뿌리)을 기반으로, 전 세계 전공자와 비전공자 개발자 단체들이 가장 쓰기 편하게 개조하여 대중화시킨 오픈소스 운영체제(OS)입니다.
* **가문의 분류**: 현재 실습 중인 환경은 대기업이나 대형 서버 환경에서 선호하는 안정적인 **'레드햇 가문 (아마존 리눅스 등)'** 이며 프로그램을 설치할 때 `dnf` 또는 `yum` 도구를 씁니다. 반면 **'데비안 가문 (우분투 등)'** 은 전 세계에서 가장 생태계가 크며 프로그램을 설치할 때 `apt`라는 도구를 씁니다. 뿌리가 같아 95%의 명령어 문법이 일치하므로, 아마존 리눅스를 완벽히 통제하면 우분투 역시 식은 죽 먹기로 다룰 수 있습니다.

## ⚡ Quick Cheat Sheet: 실전 배포 명령어 요약

실무에서 웹 파일을 AWS EC2(아마존 리눅스)에 올리고 가동할 때 사용하는 핵심 명령어 프로세스입니다.

| 단계 | 실행 명령어 (Command) | 목 적 (Purpose) |
| :---: | :--- | :--- |
| **1. 수송** | `scp -i [키페어.pem] [파일명] ec2-user@[공인IP]:~/` | 로컬 웹 파일을 AWS 개인 독방(`~`)으로 전송 |
| **2. 설치** | `sudo dnf -y install httpd` | 아파치 웹 서버 엔진 설치 |
| **3. 이동** | `sudo mv ~/[파일명] /var/www/html/` | 웹 파일을 공식 인터넷 손님 진열대로 이동 |
| **4. 권한** | `sudo chmod 644 /var/www/html/[파일명]` | 인터넷 손님이 읽을 수 있게 파일 자물쇠 해제 |
| **5. 소유** | `sudo chown apache:apache /var/www/html/[파일명]` | 파일 소유권을 아파치 요원에게 등기 이전 |
| **6. 구동** | `sudo systemctl start httpd` | 아파치 웹 서버 가동 및 서비스 시작 |

## 🚨 Troubleshooting & Trial Error (시행착오 기록)

### 01. 파워쉘 환경 변수 호출 문법 오류
* **현상**: 파워쉘 환경에서 사용자 프로필 경로를 호출할 때 구분자 오류 발생.
* **원인**: 옛날 윈도우(CMD) 방식인 `%USERPROFILE%`이나 세미콜론(`;`)을 사용하면 파워쉘 내부 엔진이 정확히 인식하지 못함.
* **해결**: 파워쉘 정석 문법인 콜론(`:`)을 활용해 **`$env:USERPROFILE`**로 호출해야 안전하고 정확하게 작동함.

### 02. `chmod` 보안 권한 조합의 미스매치
* **현상**: 파일 권한을 `614` 등으로 설정 시 아파치 웹 서버 서빙 로직에서 어색함 발생.
* **원인**: 두 번째 자리(그룹/팀원)에 실행(`1`) 권한만 주고 읽기(`4`)를 막는 것은 소스 코드 파일 관리 관점에서 매우 이례적이고 위험한 설정임.
* **해결**: 외부 손님이 읽어야 하는 웹 소스는 **`644`**(기본)로 세팅하거나, 팀원에게도 수정 권한을 주려면 쓰기(`2`)를 더한 **`664`**를 채택하는 것이 정석임.
